# 開発フロー

## 目的

初心者が迷いやすい「何から作るか」「何を仕様書に書くか」「どこまで作ったら次へ進むか」を明確にする。

この資料では、ゲーム本体を作る前に必要な制作の流れを定義する。

## 参考にする商用作品から抽象化する点

商用作品そのものを複製するのではなく、次の設計パターンを参考にする。

- コンテンツはコードに直書きせず、カード・敵・イベント・遺物をデータ化する
- 1プレイは短く、毎回違う選択が出る
- 戦闘の意思決定は「今のターンの最善」と「デッキの長期成長」の両方を問う
- プレイログを前提に、数値を頻繁に調整する

## 開発フェーズ

### Phase 0: 企画固定

期間目安: 2日から1週間

成果物:

- ゲームの一文説明
- プレイヤー体験の柱
- 作らないものリスト
- 最小スコープ

完了条件:

- 1ランの流れを文章で説明できる
- 最初の縦切り版に入れる要素と入れない要素が決まっている

### Phase 1: 紙プロトタイプ

期間目安: 1週間

成果物:

- 初期デッキ10枚
- 敵3体
- 遺物3個
- 報酬カード10枚
- 手動で回せる戦闘ルール

完了条件:

- 紙またはスプレッドシートで3戦連続プレイできる
- 1ターンにプレイヤーが何を考えるか説明できる

### Phase 2: デジタル最小戦闘

期間目安: 2週間から4週間

成果物:

- ターン開始、カード使用、敵行動、ターン終了
- 山札、手札、捨て札、廃棄
- HP、ブロック、エネルギー
- 戦闘ログ

完了条件:

- 1戦闘が最初から最後まで完了する
- カード定義をJSONなどのデータから読み込める
- バグ再現用にログを出せる

### Phase 3: 縦切り版

期間目安: 1か月から2か月

成果物:

- 1キャラ
- 20から30枚のカード
- 通常敵5体、エリート2体、ボス1体
- マップ、報酬、休憩、ショップ
- 1ステージ通しプレイ

完了条件:

- 初見プレイヤーが説明なしで1ランを終えられる
- 勝率、平均戦闘ターン、平均被ダメージを記録できる
- 重大な進行不能バグがない

### Phase 4: コンテンツ拡張

期間目安: 継続

成果物:

- 新カード
- 新敵
- 新イベント
- 新遺物
- 新キャラクター

完了条件:

- 追加コンテンツが既存のデータ形式だけで登録できる
- 調整前後の勝率と採用率を比較できる

## 週次サイクル

毎週同じリズムで進める。

| 曜日 | 作業 |
| --- | --- |
| 月 | 先週のログ確認、今週の調整テーマ決定 |
| 火 | 実装、データ追加 |
| 水 | 実装、バグ修正 |
| 木 | 内部プレイテスト |
| 金 | 数値調整、仕様更新 |
| 土 | 外部プレイテストまたは録画確認 |
| 日 | 休む、またはメモ整理だけ |

## チケットの種類

| 種類 | 内容 | 完了条件 |
| --- | --- | --- |
| Feature | 新機能 | 仕様書、実装、最低限のテストがある |
| Content | カードや敵などの追加 | データ、説明文、簡易バランス評価がある |
| Balance | 数値調整 | 調整理由と比較ログが残っている |
| Bug | 不具合 | 再現手順と修正確認がある |
| UX | 操作性・表示改善 | 変更前後のスクリーンショットまたは説明がある |
| Tech Debt | 後回しにした技術負債 | 影響範囲と削減できるリスクが明記されている |

## 作らないものリスト

縦切り版では次を作らない。

- 複数キャラクター
- 高度な実績システム
- オンライン要素
- MOD対応
- 複雑なアニメーション演出
- すべてのカードの完全バランス

初心者開発で一番危険なのは、ゲームの核が固まる前に周辺機能を作りすぎること。
