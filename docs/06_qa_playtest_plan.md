# QA・プレイテスト計画

## 目的

「なんとなく面白くない」「なんとなく難しい」で止まらないように、観察する項目を決めておく。

## テストの種類

| 種類 | 目的 | 頻度 |
| --- | --- | --- |
| Smoke Test | 起動、1戦闘、報酬、終了まで壊れていないか | 毎日 |
| Regression Test | 以前直したバグが再発していないか | リリース前 |
| Balance Test | 数値が目標に近いか | 週1 |
| UX Test | 操作や表示で迷わないか | 新画面追加時 |
| Blind Playtest | 初見プレイヤーが理解できるか | 縦切り版以降 |

## Smoke Testチェックリスト

- タイトルからラン開始できる
- 戦闘でカードを使える
- Energyが足りないカードは使えない
- 敵意図が表示される
- ターン終了できる
- 敵が行動する
- HPが0になると敗北する
- 敵を全滅させると勝利する
- 報酬を選べる
- マップへ戻れる
- ログにエラーがない

## バグ報告テンプレート


## プレイテストで見ること

観察項目:

- プレイヤーが最初にどこを見るか
- 敵の意図を理解しているか
- カードの効果を読み間違えるか
- 報酬選択に悩むか
- 何が原因で負けたと感じたか
- もう1回遊びたいと言うか

聞く質問:

- 一番楽しかった瞬間はどこですか
- 一番分かりにくかった表示はどこですか
- 負けた理由は納得できましたか
- 強いと思ったカードはどれですか
- 弱い、または取る意味がないと思ったカードはどれですか

避ける質問:

- このゲーム面白いですか
- このカード強すぎますか
- どう直せばいいですか

プレイヤーは開発者ではないため、解決策よりも反応と行動を見る。

## プレイログの読み方

1回のプレイで判断しない。

最低でも10ラン、できれば30ラン単位で見る。

優先して見る指標:

- どの階層で死ぬか
- どの敵で被ダメージが大きいか
- 取られないカードは何か
- 取りすぎるカードは何か
- 休憩で回復ばかり選んでいないか

## リリース判定

縦切り版を他人に渡してよい条件:

- 進行不能バグがない
- 1時間遊んでクラッシュしない
- 主要UIに仮テキストが残っていない
- カード説明と実効果が一致している
- ラン終了までの導線がある
- バグ報告に必要なseedとrunIdが表示できる
